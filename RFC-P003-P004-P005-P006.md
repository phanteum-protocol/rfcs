**Number:** RFC-P003

## Abstract

RFC-P003 defines **PhanosScore** — a universal, real-time trust score for AI agents computed by the Phanteum Bittensor subnet from verifiable on-chain settlement data. PhanosScore is a single floating-point number between 0.0 and 1.0. It is **free to query by anyone, forever**. The subnet earns from Bittensor TAO emissions, not from charging for queries.

---

## 1. Motivation

Every AI agent marketplace faces the cold-start trust problem: how does a buyer know if a new agent is trustworthy before hiring it? Today, agents self-report their capabilities. There is no independent verification.

PhanosScore solves this by grounding trust in **economic reality**: every score is computed from real Hedera HCS settlement records — actual HBAR exchanged for actual completed work. Scores cannot be bought or faked cheaply because fabricating a good score requires actually completing real tasks with real money.

This is the difference between a LinkedIn endorsement and a bank statement.

## 2. Score Definition

```
PhanosScore ∈ [0.0, 1.0]
```

| Range | Interpretation |
|-------|---------------|
| 0.90 – 1.00 | Elite. Top-tier agents. Enterprise-grade. |
| 0.75 – 0.89 | High trust. Reliable for most tasks. |
| 0.60 – 0.74 | Moderate trust. Suitable for low-stakes tasks. |
| 0.40 – 0.59 | Low trust. New agents or inconsistent performance. |
| 0.00 – 0.39 | Very low. Do not hire without strong due diligence. |

---

## 3. Score Computation

PhanosScore is computed by **PhanosMiner** nodes on the Phanteum Bittensor subnet (subnet ID: `phanteum`). Miners pull ground-truth data from Hedera HCS and compute scores. PhanosValidator nodes rank miners by accuracy. Yuma Consensus distributes TAO emissions to top-ranked miners.

### 3.1 Input Signals

| Signal | Weight | Source |
|--------|--------|--------|
| Task completion rate | 35% | Hedera HCS settlement records |
| On-time delivery rate (SLA adherence) | 25% | PhanosClock HCS records |
| Dispute rate (lower = better) | 20% | PhanosArbitration HCS records |
| Buyer quality ratings | 10% | VERIFY verb payload ratings |
| Task volume (log scale) | 5% | Task count from PhanosAgent registry |
| Score trajectory (improving?) | 5% | Rolling 30-day trend |

### 3.2 Update Frequency

Scores are updated every **Bittensor epoch** (360 blocks, approximately 72 minutes). New agents start at `0.50` (neutral) and scores adjust based on real activity.

### 3.3 Anti-Gaming Properties

- Ground truth = real HBAR on Hedera HCS. Cannot be faked without spending real money.
- Self-dealing (buyer and seller are the same entity) is detected by wallet graph analysis.
- Score manipulation attempts trigger PhanosSentinel review.
- After 1,000,000 real transactions, the training data moat is insurmountable.

# RFC-P004: PhanosSettle — Universal Payment Settlement Protocol

**Number:** RFC-P004  
**Title:** PhanosSettle — Atomic Escrow and Settlement for Agent Transactions  
**Status:** Draft | **Version:** 0.1.0  
**Author:** Terrence Stephens (@IamPhanos), Emerald Dawn Holdings  
**Created:** 2026-03-24 | **License:** MIT  
**Repository:** https://github.com/phanteum-protocol/rfcs  
**Depends on:** RFC-P001 (AgentDID), RFC-P002 (PhanosTask)

---

## Abstract

RFC-P004 defines **PhanosSettle** — the universal payment settlement protocol for AI agent transactions. PhanosSettle provides atomic escrow on Hedera Hashgraph with automatic toll collection. The **0.15% settlement toll** (15 basis points) is embedded in every settlement automatically — it is not an optional fee but a structural property of the protocol, similar to how TCP/IP checksum validation is not optional.

---

## 1. Settlement State Machine

```
LOCKED → ACCEPTED → RELEASED → SETTLED
                             ↓
                        REFUNDED (on dispute resolution)
                        PARTIALLY_SETTLED (partial dispute)
```

| State | Trigger | Description |
|-------|---------|-------------|
| `LOCKED` | ACCEPT verb received | Buyer HBAR locked in PhanosEscrow |
| `ACCEPTED` | PhanosEscrow confirms lock | Escrow confirmed on Hedera |
| `RELEASED` | VERIFY verb approved | Escrow release authorized |
| `SETTLED` | PhanosTollRouter executes | Funds distributed, HCS record created |
| `REFUNDED` | Arbitration: buyer wins | Full escrow returned to buyer |
| `PARTIALLY_SETTLED` | Arbitration: split | Partial payment to seller |

---

## 2. The 0.15% Toll

Every settlement passes through `PhanosTollRouter.sol` on Hedera EVM. The toll is 15 basis points (0.15%) of the gross settlement amount. It fires automatically — the seller cannot opt out.

### 2.1 Toll Distribution

```
Gross settlement: 1,000 HBAR

Toll (0.15%):           1.50 HBAR
├── EDH Treasury (60%): 0.90 HBAR → Emerald Dawn Holdings operations
├── DAO Reserve (20%):  0.30 HBAR → AIC governance treasury
├── Validators (15%):   0.225 HBAR → Bittensor validator rewards
└── $PHTM Burn (5%):   0.075 HBAR worth → burned quarterly

Net to seller:        998.50 HBAR
```

### 2.2 Toll Philosophy

The 0.15% toll is not a "protocol fee" in the traditional sense. It is a **settlement service fee** — the same way Visa charges 2.9% for settlement over a free internet. HTTP is free. Visa charges for the settlement layer on top. TCP/IP is free. Phanteum charges 0.15% for the settlement layer on top. The RFCs (P001–P006) are free forever.

---

## 3. Smart Contract Interface

### 3.1 PhanosEscrow.sol — Key Functions

```solidity
// Lock buyer funds when agent accepts a task
function lockEscrow(
    bytes32 taskId,
    address buyerHederaWallet,
    address sellerHederaWallet,
    uint256 amountHbar,
    uint256 deadlineBlock
) external returns (bytes32 escrowId);

// Release escrow after VERIFY — triggers toll deduction automatically  
function releaseEscrow(
    bytes32 taskId,
    bytes32 escrowId
) external returns (bool success);

// Emergency refund on dispute resolution
function refundEscrow(
    bytes32 taskId,
    bytes32 escrowId,
    uint256 refundPct  // 0-100
) external onlyArbitrator returns (bool success);
```

### 3.2 PhanosTollRouter.sol — Toll Logic

```solidity
uint256 constant TOLL_BPS = 15;  // 15 basis points = 0.15%

function routeToll(uint256 grossAmount) internal {
    uint256 toll = (grossAmount * TOLL_BPS) / 10000;
    
    uint256 edhShare      = (toll * 60) / 100;
    uint256 daoShare      = (toll * 20) / 100;
    uint256 validatorShare = (toll * 15) / 100;
    uint256 burnShare     = toll - edhShare - daoShare - validatorShare;
    
    // Transfer shares
    payable(EDH_TREASURY).transfer(edhShare);
    payable(DAO_RESERVE).transfer(daoShare);
    payable(VALIDATOR_POOL).transfer(validatorShare);
    // burnShare accumulated for quarterly $PHTM burn
    burnAccumulator += burnShare;
}
```

## Abstract

RFC-P006 defines **PhanosPassport** — a cryptographically-anchored compliance record for AI agents that satisfies the transparency requirements of EU AI Act Article 13. Every PhanosPassport is issued on ICP (PhanosPassport canister), anchored permanently on Hedera Consensus Service, and verifiable by any EU regulatory authority without requiring access to proprietary systems.

This RFC is submitted to ETSI TC CYBER as a candidate technical specification for the Article 13 implementing acts currently under development.

---

## 1. Regulatory Background

**EU AI Act Article 13** requires that providers of high-risk AI systems ensure those systems are sufficiently transparent to enable users to interpret output and use it appropriately. For AI agents — which autonomously take actions with legal and financial consequences — Article 13 requires:

- Identity declaration (who/what is this agent)
- Capability declaration (what can this agent do)
- Limitations declaration (what can this agent NOT do)
- Audit trail (what has this agent done)
- Human oversight mechanism (how can a human intervene)

PhanosPassport addresses all five requirements in a single, machine-readable, cryptographically-verifiable record.
